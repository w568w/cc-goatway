# 编码风格 / 重构指导

- 核心目的：在保证最佳实践的前提下，减少代码行数，提高可读性。
- 锁、并发、异步相关的代码要格外注意，保持清晰，清楚地说明每个锁的目的、范围和使用方式。
- 不要添加无意义的包装函数，尤其是那些只是简单调用另一个函数的函数，或者只被调用一次的函数。
- 将相似的逻辑聚合在附近，保持空间局部性，避免过度分散。
- 警惕「巨型状态结构体」。如果一个结构体包含了过多的字段，考虑将其拆分成更小、更专注的结构体，或者严格地考虑每个字段的必要性，添加注释说明为什么需要它们，将相关的字段聚合在一起。
- 同样地，警惕包含太多函数的「巨型模块/结构体」。
- 对于每一个功能项，选择适合的数据结构和算法抽象，保持代码的清晰性。
- 如果一个函数充斥的没有逻辑的代码（如大量的 if-else 嵌套、超长的判断链、访问几十个字段的代码），说明你的设计有问题，函数应该更专注于一个单一的功能，或者你需要引入新的抽象来简化代码。
- 重构中不要随便拆分文件/结构体。拆分不降低复杂度，反而可能增加复杂度，需要在多个实体之间跳转才能理解代码。在一个文件中提高代码的清晰度和可维护性，而不是通过拆分文件来掩盖复杂性。
- 简洁和可读性是第一要务。写完后审查一下是否有冗余或不清晰的部分。
- 不允许擅自添加自己认为需要的功能。除非用户明确要求，否则不要添加任何新的功能或修改现有功能。你可以提出建议，但必须先得到用户的确认。
- 充分使用类型系统，用正确的类型来表达你的意图，避免使用过于宽泛的类型。尤其避免：
  - 字符串滥用：例如，当比较 IP 时，使用专门的 IP 类型而不是字符串；
  - 空值滥用：在需要严格区分 unset、null、empty 的场景中，使用指针类型来表达这些不同的状态，而不是使用零值混为一谈。

## 复刻指导

- 复刻时，不得擅自编写在用户要求之外的代码。即使有你认为的错误代码，也应当首先向用户确认是否需要修复。
- 不需要完全照搬原本的函数和模块设计，保持简洁。
- 必须保持功能的一致性，除非用户明确要求修改功能。
- 复刻时，保持代码的清晰和可读性，不要逐行翻译。思考在目标语言中有没有更合适的表达方式。
